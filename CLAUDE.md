# CLAUDE.md — VLUX Phase 1

## プロジェクト概要

VLUX（ヴルクス）は整骨院向けAI搭載ヘルスケアSaaSプラットフォーム。
INDEX CorporationがRYOJ Inc.（馬場竜次代表）からPhase 1開発を受託。

スタック構成：
- フロントエンド：React / PWA（患者スマホ・院内iPad共通）
- バックエンド：Supabase（PostgreSQL・Auth・Storage・Realtime・Edge Functions）
- AIエンジン：Claude API（カルテ生成・生活アドバイス生成）
- 音声入力：Web Speech API（日本語）
- ヘルスデータ：Apple HealthKit / Google Fit（PWA経由・基本値のみ）
- 通知：Web Push / FCM
- デプロイ：Vercel
- リポジトリ：GitHub（プライベート）

Phase 1の対象は2〜3医院のクローズドβ。
EC・クーポン・動的ランク・決済はPhase 2以降。

このコードベースは本番稼働する医療関連システムです。
全ての変更は患者データの完全性・プライバシー保護・診療業務への影響を考慮して行うこと。

## コア原則

### 1. 患者データの安全を最優先
以下には慎重に扱う：
- 患者の個人情報・診療データの読み書きロジック
- Supabase RLS（行レベルセキュリティ）設定
- JWT認証・セッション管理
- 音声データの録音・保存・削除
- HealthKitデータの取得・同期
- Claude APIへ送信するプロンプト（PHI混入に注意）

これらに触れる場合は必ず：
1. リスクを明示する
2. 最小限の安全な変更を提案する
3. 副作用を確認する
4. 検証手順を示す

### 2. 実装前に考える
小さな修正以外は、即コードに入らない。
- 目的を明確にする
- 関連ファイルを調べる
- 現状の挙動を説明する
- 失敗パターンを洗い出す
- 方針を提案してから実装する

### 3. 差分を最小に保つ
不要なリファクタリングは避ける。
小さく・可逆性があり・スコープが明確な編集を優先する。

### 4. システム全体の整合性を保つ
このプロジェクトは以下が密結合している：
- Supabase RLSポリシー
- Edge Functions（API層）
- フロントエンドコンポーネント（院側 / 患者側）
- Claude APIプロンプト
- HealthKit / Google Fit連携フロー
- Push通知フロー

一つのファイルを変更する前に、下流の依存関係を確認する。

### 5. 本番を意識する
このシステムは実際の診療現場で動く。
以下を優先して最適化する：
- 正確性
- 観測可能性（ログ・エラー表示）
- ロールバック安全性
- デバッグ容易性
- 操作の簡潔性

## 作業スタイル

機能追加・バグ修正を依頼された場合、以下の順番で動く：

1. 関連コードを先に読む
2. 現在の動作を要約する
3. リスク・矛盾点を洗い出す
4. 実装方針を提案する
5. コードを編集する
6. 編集後に以下をまとめる：
   - 変更内容
   - 変更理由
   - 起こりうる副作用
   - テスト方法

非自明なタスクでは、計画ステップを飛ばさない。

## レビュー観点

コードレビュー時は以下を優先する：

- 患者データの漏洩・混在リスク
- RLSポリシーの抜け穴
- Claude APIへのPHI（要個人情報）混入
- 音声データの未削除・不正保存
- HealthKit認証フローのエッジケース
- Supabase接続エラーのハンドリング
- 重複カルテ生成のリスク
- Push通知の重複送信
- 非同期処理の競合状態
- エラーのサイレント失敗
- 院側・患者側UIの仕様不一致
- 将来のマルチテナント化への影響

浅い称賛は不要。直接的・具体的・技術的に正直であること。

## 編集ルール

### やること
- スコープを絞った変更を行う
- 既存の命名規則を維持する
- リスクのあるロジックにはガードを追加する
- 有用な場所のみログを追加する
- 既存アーキテクチャを尊重する（強い理由がない限り）

### やらないこと
- 正当な理由なくファイル全体を書き直す
- 不明なビジネスルールを勝手に実装する
- ログを気軽に削除する
- Supabase環境変数名を気軽に変更する
- APIペイロードのセマンティクスを変更するのに指摘しない
- 動作が不明なのに確信があるように振る舞う

不確かなことがあれば、率直にそう言う。

## テスト方針

変更を加えた際は必ず具体的な検証手順を示す。
以下のような実践的なテストを優先する：

- Supabase RLS動作確認（患者Aから患者Bのデータが見えないか）
- Claude API呼び出しのモックテスト（プロンプト・レスポンス確認）
- Web Speech API音声入力の実機テスト（iPhoneまたはiPad）
- HealthKit同期の動作確認（実機必須・シミュレーター不可）
- Push通知の動作確認（iOS / Android 別々に）
- 認証フロー（OTPメール・SMS）のテスト
- 院側→患者側へのカルテ送信フローの結合テスト
- エラーパス（API失敗・ネットワーク切断）のログ確認

自動テストが存在しない場合は、手動確認の手順を明示する。

## アーキテクチャの考え方

以下のメンタルモデルに従う：

- Web Speech API = 音声入力エミッター
- Edge Functions / Supabase = 決定・実行の制御層
- Claude API = 外部依存（失敗する可能性がある）
- 患者PWA = データ受信・表示サーフェス
- 院内iPad UI = 音声入力・カルテ確認サーフェス（ソースオブトゥルースではない）
- Supabase DB = 単一の信頼できるデータ源
- ログ = 重要な運用記録

変更の提案はこれらの境界線を尊重すること。

## セキュリティルール（重要）

### Rule 1: RLSを必ず確認する
Supabaseで新しいテーブルやクエリを追加する場合：
- 必ずRLSポリシーを設定する
- 院間のデータ分離が保たれているか確認する
- 患者A→患者Bのデータアクセスが不可能であることを確認する

### Rule 2: Claude APIへのPHI混入を防ぐ
プロンプトを設計・変更する場合：
- 氏名・生年月日・住所などの個人情報をAPIに直接送らない
- 患者IDや匿名化済みキーを使う
- プロンプトのログに個人情報が混入しないか確認する

### Rule 3: 音声データの適切な管理
音声録音機能に触れる場合：
- 録音データの保存先・暗号化を確認する
- 使用後の削除フローを確認する
- 患者の同意取得フローと紐付いているか確認する

### Rule 4: 認証・セッション管理
認証ロジックに触れる場合：
- JWTの有効期限を確認する
- リフレッシュトークンの安全な管理を確認する
- 不正アクセス時のエラーハンドリングを確認する

### Rule 5: エラーのサイレント失敗禁止
常に：
- エラーを明示的に throw する
- ログに記録する
- デバッグのための情報を残す

### Rule 6: 不確かな時は何もしない
不確実性がある場合：
- 処理をスキップする
- 理由をログに記録する
- 確認を求める

## 開発者について（重要）

このプロジェクトの開発担当者（かっちゃん）は、文系バックグラウンドのビジネスパーソンです。
プログラミング・サーバー・データベースの専門知識はありません。

これを前提に、以下を必ず守ること。

### 説明のルール

**専門用語を使う場合は必ず一言で補足する**
- 悪い例：「RLSを設定してください」
- 良い例：「RLS（データベースのアクセス制限ルール）を設定します」

**何をしているのか、なぜそれをするのかを常に説明する**
- コードを書くだけでなく、「このコードは〇〇をするために書いています」と添える
- エラーが出たら「なぜそのエラーが出たのか」を日本語で先に説明する

**手順は細かく・1ステップずつ**
- 「〇〇して、次に△△して、最後に□□してください」のような連続指示は避ける
- 1つ作業が終わったら確認を取り、次に進む

**「コピーして貼り付けるだけ」で動くように書く**
- 変数名や設定値をどこに何を入れればいいか、具体的に示す
- 「適宜変更してください」という曖昧な指示は避ける

**エラーが出たときのメッセージは怖くない**
- エラー文をそのまま見せるだけでなく、「このエラーは〇〇が原因です。以下の手順で直せます」と案内する

### 作業確認のルール

- 大きな変更の前に「これから〇〇をします。よろしいですか？」と確認する
- 完了したら「〇〇ができました。次は△△に進みます」と現在地を伝える
- うまくいかない場合は「何が起きているか」「どうすれば直るか」を丁寧に説明する

### やってはいけないこと

- 専門用語の羅列だけで終わる説明
- 「後は適当に設定してください」「必要に応じて変更してください」などの丸投げ
- エラー文の貼り付けだけで原因説明なし
- 複数の作業を一度に指示する（消化不良になる）
- 「わかりますよね？」を前提にした説明

## 辞書機能実装サマリー（Phase 1）

### 概要
AIカルテ生成パイプラインの精度を高めるため、3層構造の辞書機能を実装。
現場データが蓄積されるにつれてバージョンアップしていく運用設計。

---

### 第1層：Whisper APIプロンプト辞書
音声→テキスト変換の精度を上げるための辞書。

**実装済みカテゴリ：**
- 医療用語辞書（筋肉名・骨関節名・症状名・施術用語・鍼灸ツボ名）
- 生活習慣・職業ワード辞書
- 期間・頻度の表現辞書
- 感覚表現・擬音語・擬態語辞書
- 九州5エリア方言辞書

**目的：** Whisperが「りしょうきん→梨状筋」「だれた→疲れた」と正確に変換できるようにする。

---

### 第2層：Claude Haikuプロンプト辞書
テキスト→構造化JSONの精度を上げるための辞書。

**実装済みカテゴリ：**
- 九州5エリア方言辞書 v0.3（現代版・現場確認済み）

**目的：** 方言が混入した会話テキストを正しく構造化する。

---

### 第3層：Claude Sonnetプロンプト辞書
構造化JSON→カルテ生成の精度を上げるための辞書。

**実装済みカテゴリ：**
- 九州5エリア方言辞書 v0.3（現代版・現場確認済み）

**目的：** 方言を正確に標準語に補完してSOAPカルテを生成する。

---

### 九州方言辞書 v0.3（確定版）
対象エリア：鹿児島・博多・長崎・久留米・北九州

**鹿児島弁（現地ネイティブ確認済み）：**
- だれた → 疲れた
- しびるっど → しびれる
- 痛か → 痛い
- かて → 硬い・動かしにくい
- わっぜ → とても・すごく
- ちっとばっかい → 少し
- ここんとこずっと → 最近ずっと

**博多弁：**
- いたかとー → 痛いよ
- だるかばい → だるいよ
- きつかとー → しんどいよ
- しびれとう → しびれている
- よれとう → 疲れ果てている
- ばりばり → とても・すごく

**長崎弁：**
- そうたい → そうだよ
- ほんとたい → 本当だよ
- がばい → すごく
- ぼちぼち → まあまあ
- なおらん → 治らない

**久留米弁：**
- いたかばい → 痛いよ
- だるかばい → だるいよ
- ずっとばい → ずっとだよ
- よかよか → いいよ・大丈夫

**北九州弁：**
- いたかっちゃん → 痛いんだよ
- そうっちゃん → そうなんだよ
- たいぎゃ → とても・大変
- ばりばり → とても・すごく

---

### 運用方針
```
現在：v0.3（現場確認済みの鹿児島弁ベース）
↓
βテスト中：誤変換・漏れを記録
↓
v0.4・v0.5と随時更新
↓
展開エリア拡大時：対応方言を追加
```

**対象外（Phase 2以降）：**
- 九州外の方言（大阪弁・広島弁・東北弁など）
- 理由：Phase 1の対象クリニックが九州エリアのため
